# 项目介绍 — RAG 电商导购 Agent

> 用于面试时快速介绍项目，30 秒电梯演讲 + 2 分钟详细版本

---

## 一、30 秒电梯演讲

> 我做了一个基于 RAG 的电商导购 Agent，用户上传图片或输入文字，系统通过 CLIP 图像特征 + BGE-M3 文本特征做混合检索，千问 gte-rerank-v2 精排后交给 DeepSeek 生成带引用的购物推荐。200 个 SKU 测试中，同款识别 Top-1 达到 100%，Faithfulness 0.82。工程上用了 LangGraph 做 Agent 编排、Redis 做会话和状态持久化、Qdrant 做向量存储，还实现了多租户架构支持多团队共用后端。

## 二、2 分钟详细介绍

### 项目背景
这是一个毕设项目，目标是做一个能理解图片和文字的电商导购助手。用户拍一双鞋的照片，系统能找出同款或相似商品，并给出推荐理由。

### 技术架构

```
用户输入（图片/文字）
    ↓
Embedding 层
├── CLIP → 512 维图像向量
└── 阿里云 text-embedding-v3 → 1024 维文本向量
    ↓
Qdrant 混合检索
├── 图像 top-k（score_threshold=0.6）
└── 文本 top-k（score_threshold=0.5，支持 category filter）
    ↓
RRF 融合（Reciprocal Rank Fusion）
    ↓
千问 gte-rerank-v2 精排
    ↓
结构化过滤（price filter）
    ↓
LangGraph Agent 编排
├── 槽位填充（提取预算/品类/偏好）
├── 检索节点 → 候选商品
├── 引用节点 → citation 片段
├── 澄清节点 → 多因子门控（低置信/难分伯仲/槽位不完整）
└── 生成节点 → DeepSeek 流式输出（显式引用 + 不确定表达）
    ↓
SSE 推送前端（delta_text / candidates / citations / final）
```

### 核心功能

1. **混合检索**：CLIP 图像 + BGE-M3 文本，RRF 融合，Top-3 命中率 69%
2. **Rerank 精排**：千问 gte-rerank-v2，对 query-候选对精排
3. **槽位填充**：规则提取 + LLM 兜底，提取预算/品类/场景/偏好
4. **结构化过滤**：category filter（Qdrant）+ price filter（检索后过滤）
5. **多因子澄清**：top1 score + gap + 槽位完整度综合判断
6. **答案增强**：显式引用商品名称 + 不确定表达 + 推荐解释
7. **多租户架构**：X-Tenant-ID 中间件 + Redis key 隔离 + 配额管理
8. **四层记忆**：Working/Episodic/Semantic/Perceptual，Redis 持久化
9. **LLM 治理**：重试 + 熔断 + fallback（DeepSeek → 通义千问）+ 限流

### 核心指标

| 指标 | 数值 | 说明 |
|------|------|------|
| Top-1 同款识别 | 100% | CLIP 对同款商品区分度极高 |
| Top-3 hybrid | 69% | 比纯文本模式提升约 3.5 个百分点 |
| Faithfulness | 0.82 | 生成内容忠实于检索上下文 |
| Context Precision | 0.82 | 检索内容相关性高 |
| P95 延迟 | ~3s | 电商导购场景可接受 |
| SKU 规模 | 200 商品 | 跨品类 PoC |

## 三、技术亮点（面试重点）

### 1. 混合检索 + Rerank

**问题**：纯文本检索对"找同款"场景效果差，纯图像检索对"找相似但不同款"效果差。

**解决**：
- CLIP 图像向量 + BGE-M3 文本向量，各自检索 top-k
- RRF 融合：`score(d) = Σ 1/(k + rank_i(d))`，k=60
- 千问 gte-rerank-v2 精排，对 query-候选对成对打分

**效果**：Top-3 从 65% 提升到 69%

### 2. 槽位填充 + 结构化过滤

**问题**：用户说"推荐一双 500 元以内的篮球鞋"，纯语义检索会把"500 元"当普通文本。

**解决**：
- 规则提取：正则匹配预算（500元以内 / 预算500 / 300-500）
- LLM 兜底：复杂表达时调用 LLM 提取结构化信息
- 结构化过滤：category filter 传给 Qdrant，price filter 检索后过滤

**效果**：检索更精准，减少无关商品

### 3. 多因子澄清门控

**问题**：单一阈值（top_score < 0.6）导致过度澄清或澄清不足。

**解决**：
```python
if top_score < 0.6:
    强澄清（低置信）
elif top_score < 0.75 and gap < 0.05 and not (预算+品类):
    弱澄清（难分伯仲 + 槽位不完整）
else:
    不澄清
```

**效果**：减少误触发，澄清问题更精准

### 4. 多租户架构

**问题**：多个团队/店铺想共用同一套后端，但数据要隔离。

**解决**：
- X-Tenant-ID 中间件：解析 + 校验 + 注入 request.state
- Redis key 隔离：`agent:tenant:{tenant_id}:session:{sid}`
- 配额管理：请求数/token 数限制，Redis INCR + TTL

**效果**：一套后端服务多个租户，数据完全隔离

### 5. 四层记忆 + Redis 持久化

**问题**：会话在进程内存，重启丢失；多副本无法共享会话。

**解决**：
- Working Memory：LangGraph state（当前 run 的瞬时状态）
- Episodic Memory：会话历史（Redis Hash，7 天 TTL）
- Semantic Memory：用户偏好（Redis，跨会话持久化）
- Perceptual Memory：图像/文本向量（Qdrant）

**效果**：支持多轮对话、跨会话偏好、多副本共享

## 四、面试高频问题

### Q1: 为什么选 CLIP + BGE-M3，而不是多模态大模型？
**A**: 三个原因：
1. **成本**：多模态 API 调用成本是 embedding 的 100 倍以上
2. **可控性**：RAG 可以显式控制召回 + 加 citation，多模态是黑盒
3. **可扩展**：商品库更新只需重新 embedding，不需要重新训练

### Q2: RRF 融合的 k 为什么选 60？
**A**: k 是平滑常数，防止排名靠前的文档分数差距过大。k=60 是经验值，来自原始论文。k 越大，排名影响越平滑；k 越小，头部文档优势越明显。我们没有调参，直接用默认值。

### Q3: 为什么槽位填充用"规则 + LLM 兜底"？
**A**: 
- **规则提取**：快速（<10ms）、确定性高、不消耗 token
- **LLM 兜底**：处理复杂表达（"我想找一双适合宽脚的篮球鞋，预算不超过 600"）
- **成本平衡**：只在规则提取失败时调用 LLM，节省成本

### Q4: 多租户为什么用 Header 而不是 JWT？
**A**: MVP 阶段简化实现。Header 方案对前端调试友好，后续可叠加 JWT，不冲突。

### Q5: Redis 挂了怎么办？
**A**: 
- SessionManager 降级为内存模式（但无法跨进程共享）
- TenantManager 同理
- LLM 调用有熔断 + fallback（DeepSeek → 通义千问）

### Q6: Faithfulness 0.82 算高吗？
**A**: 在电商导购场景，0.82 说明 82% 的生成内容有检索上下文支撑。剩余 18% 可能是推荐话术和总结性描述，这部分不需要 citation 支撑。实际有事实断言的准确率 > 90%。

### Q7: Context Recall 为什么只有 0.38？
**A**: 这是因为 citation 切分粒度粗（整块 introduction），ground truth 中的细节可能没被覆盖。我们加了"每候选至少一条引用"的兜底，但还没重测。

### Q8: 如果商品库从 200 SKU 扩展到 10 万 SKU，你会怎么改？
**A**:
1. **Embedding**：批量化处理，用 GPU 加速 CLIP 推理
2. **Qdrant**：从 Cloud 迁移到自建集群，开 HNSW 索引优化
3. **Rerank**：改为本地模型（BGE-reranker），避免 DashScope 按量计费成本爆炸
4. **缓存**：对热门 query 的检索结果做 Redis 缓存
5. **异步化**：embedding 和 rerank 改为异步队列，避免阻塞请求
