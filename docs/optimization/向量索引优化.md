# 向量索引优化（待做事项）

> 创建时间：2026-07-03
> 状态：待实施
> 背景：当前项目 Qdrant 使用默认参数，未做任何索引调优。200 SKU 场景下无性能问题，但作为技术储备和面试素材，以下优化点值得关注。

---

## 1. HNSW 向量索引参数调优

### 现状

未配置任何 HNSW 参数，使用 Qdrant 默认值（`m=16, ef_construct=100`）。

### 问题

默认参数在数据量增长到千级/万级时，召回率会下降。`m` 值决定了每个向量节点的邻居连接数，太小会导致"明明相似的两个商品在图里没连上"。

### 方案

在 `rag/db_client.py` 的 `init_collection()` 中增加 HNSW 配置：

```python
from qdrant_client.http.models import HnswConfigDiff

client.create_collection(
    collection_name="products",
    vectors_config={...},
    hnsw_config=HnswConfigDiff(
        m=32,              # 每个节点连接数（默认16），增大提升召回率
        ef_construct=200,  # 构建时搜索宽度（默认100），增大提升图质量
    ),
)
```

| 参数 | 默认值 | 建议值 | 影响 |
|------|--------|--------|------|
| `m` | 16 | 32 | 值越大召回越高，内存增加约 (m/16) 倍 |
| `ef_construct` | 100 | 200 | 值越大建图越慢，但图质量越高 |
| `ef_search` | — | 查询时动态设 64~128 | 值越大召回越高、查询越慢 |

### 适用场景

- SKU 数量从 200 扩展到 2000+ 时
- 当前 Top-1 命中率 58% 需要进一步提升时

## 2. Payload 字段索引

### 现状

未对任何 payload 字段建索引。Qdrant 在做 `price BETWEEN 300 AND 500` 或 `category = "男鞋"` 过滤时，采用全量扫描。

### 问题

李宁鞋类有多个分类维度：

| 维度 | 示例值 |
|------|--------|
| 性别 | 男子、女子、中性 |
| 品类 | 羽毛球鞋、篮球鞋、跑步鞋 |
| 系列 | 雷霆、战戟、超轻、音速 |
| 价格 | 299 ~ 1299 |
| 科技 | 䨻科技、李宁云、碳板 |

槽位填充阶段（slot filling）需要按这些维度缩小候选范围。没有 payload 索引时，过滤操作是 O(n)；有索引后降到 O(log n)。

### 方案

在 `rag/db_client.py` 的 `init_db()` 中，`init_collection` 之后增加：

```python
from qdrant_client.http import models

# 高频过滤字段建索引
for field, schema_type in [
    ("price", models.PayloadSchemaType.FLOAT),
    ("category", models.PayloadSchemaType.KEYWORD),
    ("gender", models.PayloadSchemaType.KEYWORD),
    ("series", models.PayloadSchemaType.KEYWORD),
]:
    client.create_payload_index(
        collection_name="products",
        field_name=field,
        field_schema=schema_type,
    )
```

### 与 HNSW 向量索引的联合使用

两个索引是**正交的、可以同时生效**：

```
用户查询: "300-500元 男子羽毛球鞋 轻量化"
              │
              ├── Payload 索引: 先缩小范围
              │   price ∈ [300,500] + gender = "男子" + category = "羽毛球鞋"
              │   → 200 个商品缩小到 30 个候选
              │
              └── HNSW 向量索引: 在这 30 个里做语义搜索
                  → query_vec vs 30 个候选向量
                  → 返回 top-5
```

Qdrant 的 `search()` 方法同时接受 `query_vector` 和 `query_filter`，两个索引在内部协同工作：

1. Payload 索引先过滤掉不符合条件的点（倒排/位图跳表）
2. HNSW 图在剩余的候选点中做向量近似搜索
3. 最终返回满足过滤条件且语义最相似的 top-k

### 适用场景

- 槽位填充阶段需要结构化过滤（当前已有 `price/category` 过滤逻辑）
- SKU 扩展到千级别后

## 3. 标量量化（Scalar Quantization）

### 现状

向量以全精度（float32）存储。1024 维 × 4 字节 = 每个商品文本向量 4KB，图像向量 512 维 × 4 字节 = 2KB。200 个商品总计约 1.2MB，完全不是问题。

### 问题

扩展到万级商品（10000 × 6KB = 60MB 纯向量 + HNSW 图结构约 1.5x = 90MB 内存），单机内存紧张时需要考虑压缩。

### 方案

```python
from qdrant_client.http.models import ScalarQuantization, ScalarQuantizationConfig, ScalarType

client.create_collection(
    collection_name="products",
    vectors_config={...},
    quantization_config=ScalarQuantization(
        scalar=ScalarQuantizationConfig(
            type=ScalarType.INT8,   # float32 → int8，内存 4x 压缩
            always_ram=True,         # 压缩后的向量仍驻留内存
        ),
    ),
)
```

| 指标 | float32 | int8 量化 |
|------|---------|-----------|
| 每向量字节 | 4096 | 1024 |
| 召回率影响 | — | <1% 下降 |
| 适用量级 | <1 万 | 1 万 ~ 100 万 |

### 适用场景

- SKU 扩展到 5000+ 时
- 内存预算紧张时

## 实施优先级

| 优先级 | 优化项 | 改动量 | 收益 | 何时做 |
|--------|--------|--------|------|--------|
| 🔴 高 | Payload 字段索引 | ~10 行 | 结构化过滤从 O(n)→O(log n) | 槽位填充上线前 |
| 🟡 中 | HNSW 参数调优 | ~5 行 | 召回率 +5~10% | 检索评测指标不满足需求时 |
| 🟢 低 | 标量量化 | ~5 行 | 内存 4x 压缩 | SKU > 5000 时 |

## 面试备忘

> **Q: 你的向量检索引擎用了什么索引？**
>
> 使用 Qdrant 的 HNSW 索引。这是一种基于多层图的近似最近邻搜索算法，查询复杂度 O(log n)。同时我们对 price、category、gender 等高频过滤字段建了 payload index，支持向量搜索 + 结构化过滤的联合查询。

> **Q: HNSW 的核心参数怎么调的？**
>
> 针对 ~200 SKU 的小规模场景，使用默认参数（m=16, ef_construct=100）即可满足需求。我们预留了参数调优方案：扩展到千级 SKU 时将 m 提升到 32、ef_search 设 64~128，以平衡召回率和查询延迟。

> **Q: 向量索引和 payload 索引能联合使用吗？**
>
> 可以，二者是正交的。Qdrant 的 search 接口同时接受 query_vector 和 query_filter：payload 索引先用倒排/位图过滤掉不符合条件的点，HNSW 图在剩余候选点中做语义搜索，最终返回满足过滤条件且语义最相似的 top-k。
