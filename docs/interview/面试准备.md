## 一、项目电梯演讲（30 秒版本）

> 我做了一个基于 RAG 的电商导购 Agent，用户上传商品照片或输入文字描述，系统通过 CLIP 图像特征 + BGE-M3 文本特征做混合检索，千问 gte-rerank-v2 精排后交给 DeepSeek 生成带引用的购物推荐。50 个 SKU 跨品类测试中，同款识别 Top-1 达到 100%，Faithfulness 0.82。工程上用了 LangGraph 做 Agent 编排、Redis 做会话和状态持久化、Qdrant 做向量存储，后端 FastAPI + SSE 流式输出。

## 二、技术架构（面试官会追问的点）

### 2.1 数据流

```
用户输入（图片/文字）
    ↓
embedding 层
├── CLIP → 512 维图像向量
└── BGE-M3 / text-embedding-v3 → 1024 维文本向量
    ↓
Qdrant 混合检索
├── 图像 top-k（score_threshold=0.6）
└── 文本 top-k（score_threshold=0.5）
    ↓
RRF 融合（Reciprocal Rank Fusion）
    ↓
千问 gte-rerank-v2 精排
    ↓
LangGraph Agent 编排
├── 检索节点 → 候选商品
├── 引用节点 → citation 片段
├── 澄清节点 → 置信度低时追问
└── 生成节点 → DeepSeek 流式输出
    ↓
SSE 推送前端（delta_text / candidates / citations / final）
```

### 2.2 为什么选这些组件？（面试必问）

| 组件               | 选型                    | 为什么这么选                        | 有什么替代方案                                  |
| ---------------- | --------------------- | ----------------------------- | ---------------------------------------- |
| **图像 Embedding** | CLIP (ViT-B/32)       | 图文对齐能力强，512 维够轻量              | SigLIP, BLIP, 多模态大模型直接出 embedding        |
| **文本 Embedding** | 阿里云 text-embedding-v3 | 中文效果优于 BGE-M3，1024 维          | OpenAI text-embedding-3, Cohere embed-v3 |
| **Rerank**       | 千问 gte-rerank-v2      | 中文电商场景效果好，DashScope 按量付费      | BGE-reranker-v2-m3 本地部署，Cohere rerank    |
| **向量库**          | Qdrant Cloud          | 开源，支持 hybrid search，Cloud 免运维 | Milvus, Weaviate, Pinecone, Chroma       |
| **Agent 框架**     | LangGraph             | 状态机可控，支持 checkpointer         | LangChain Agent, AutoGen, 手写状态机          |
| **LLM**          | DeepSeek              | 性价比极高，中文好                     | GPT-4o-mini, 通义千问, GLM-4                 |
| **会话存储**         | Redis                 | 低延迟，支持 TTL，checkpointer 友好    | PostgreSQL, 内存（演示阶段）                     |

### 2.3 RRF 融合公式

```
RRF_score(d) = Σ 1/(k + rank_i(d))
```

- `k=60` 是常数，防止排名靠前的文档分数差距过大
- `rank_i(d)` 是文档 d 在第 i 路检索中的排名
- 优点：不需要归一化不同检索器的分数，直接按排名融合

## 四、高频面试问题预演

### Q1: RAG 和直接 LLM 有什么区别？为什么要用 RAG？

> RAG 解决的是 LLM 知识截止和幻觉问题。电商导购场景需要基于最新商品库回答，LLM 训练数据不包含这些 SKU。RAG 把商品信息作为上下文注入，让 LLM 基于事实生成，可以加 citation 追溯来源。直接 LLM 的话，它会编造不存在的商品。

### Q2: 为什么不直接用多模态大模型（GPT-4o/Gemini）做图文理解？

> 三个原因：
> 
> 1. **成本**：多模态 API 调用成本远高于 embedding + rerank。50 个 SKU 每次请求调一次多模态，成本是 embedding 的 100 倍以上
> 2. **可控性**：RAG 可以显式控制召回哪些商品、加 citation，多模态大模型是黑盒
> 3. **可扩展**：商品库更新只需重新 embedding，不需要重新训练模型

### Q3: Faithfulness 怎么算的？

> RAGAS 框架的计算方式：
> 
> 1. 让 LLM 把生成的 answer 拆成若干 statements（断言）
> 2. 对每个 statement，检查能否从 contexts 中找到支撑证据
> 3. Faithfulness = 有支撑的 statements 数 / 总 statements 数
> 
> 0.82 意味着 82% 的生成内容有检索`上下文`支撑。

### Q4: 向量检索的瓶颈在哪？怎么优化？

> 当前瓶颈是**近似替代场景** Top-3 只有 40%，根因是 BGE-M3 向量区分度不足（同类商品描述太相似）。
> 
> 优化方案（按成本递增）：
> 
> 1. **Rerank 精排**：已实现，cross-encoder 模型对 query-doc 对做精细打分
> 2. **优化产品描述**：在 product description 里加关键区分词（科技名称、适用场景）
> 3. **结构化过滤**：在 Qdrant 检索时加 price/category filter，缩小候选集
> 4. **Fine-tune embedding 模型**：用商品对的相似度标注数据微调 BGE-M3

### Q5: LangGraph 和 LangChain Agent 有什么区别？

> LangChain Agent 是 ReAct 循环（Thought → `Action `→ Observation），状态不可控。
> LangGraph 是显式状态机，每个节点有明确的输入输出，可以：
> 
> 1. **条件分支**：根据检索置信度决定是否走澄清节点
> 2. **循环**：反思节点可以把不满意的生成结果送回重新生成
> 3. **Checkpointer**：每步状态持久化到 Redis，服务重启可恢复
> 4. **可视化**：图结构清晰，方便调试

### Q6: Redis checkpointer 怎么实现的？

> 自实现了 `RedisSaver(BaseCheckpointSaver)`：
> 
> - `put()`：把 LangGraph state 序列化后写入 Redis，key = `agent:checkpoint:{thread_id}:{checkpoint_id}`
> - `get_tuple()`：按 thread_id 找最新的 checkpoint，反序列化恢复 state
> - 用了 Redis Sorted Set 按时间戳排序，支持多版本 checkpoint
> - Redis 不可用时降级到 InMemorySaver，避免启动失败

### Q7: 你遇到最难的技术问题是什么？

> **Rerank 的延迟和成本平衡**。千问 gte-rerank-v2 效果好但按 token 计费，最初我是对所有候选都 rerank，延迟 P95 超过 5s。
> 
> 解决方案：
> 
> 1. 先做 RRF 粗排，只保留 top-10 候选
> 2. 对 top-10 做 rerank，取 top-3 给 LLM
> 3. 这样 rerank 延迟从 3s 降到 800ms，成本降低 70%
> 
> 这个经验让我理解了**检索链路的分层设计**：先用低成本方法粗筛，再用高精度方法精排。

### Q8: 如果商品库从 50 SKU 扩展到 10 万 SKU，你会怎么改？

> 1. **Embedding**：批量化处理，用 GPU 加速 CLIP 推理
> 2. **Qdrant**：从 Cloud 迁移到自建集群，开 HNSW 索引优化
> 3. **Rerank**：改为本地模型（BGE-reranker），避免 DashScope 按量计费成本爆炸
> 4. **缓存**：对热门 query 的检索结果做 Redis 缓存
> 5. **异步化**：embedding 和 rerank 改为异步队列，避免阻塞请求

## 七、数据库深度问题（面试官必问）

### 7.1 Qdrant 向量数据库

#### Q: 为什么选 Qdrant 而不是 Milvus/Weaviate？

> **三个原因**：
> 
> 1. **开源 + 轻量**：Qdrant 是 Rust 写的，单二进制文件，部署简单
> 2. **命名向量（Named Vectors）**：原生支持多向量（text + image），适合混合检索
> 3. **Cloud 免运维**：官方提供 Cloud 版本，自动副本、故障转移，适合 PoC
> 
> **对比**：
> 
> - Milvus：功能强大但重，部署复杂（需要 etcd、MinIO、Pulsar）
> - Weaviate：GraphQL 接口，学习成本高
> - Pinecone：全托管但贵，不适合学习

#### Q: Qdrant 的 HNSW 索引原理？

> **HNSW（Hierarchical Navigable Small World）**：
> 
> - 分层图结构：顶层稀疏（快速定位），底层密集（精确搜索）
> - 查询时从顶层开始，逐层向下搜索，类似跳表
> - **关键参数**：
>   - `m=16`：每层最大连接数（越大越准，但越慢）
>   - `ef_construct=100`：构建时搜索范围（越大索引越好，但构建越慢）
>   - `ef=128`：查询时搜索范围（越大越准，但越慢）
> 
> **调优经验**：
> 
> - 默认参数适合大多数场景
> - 如果召回率低，增加 `ef`（查询时搜索范围）
> - 如果构建慢，降低 `ef_construct`

#### Q: Qdrant 如何支持多租户隔离？

> **方案 B：Payload Filter（共享集合 + 过滤）**
> 
> - 所有租户数据存在同一个 collection
> - 每个向量带 `tenant_id` 字段
> - 检索时加 filter：`Filter(must=[FieldCondition(key="tenant_id", match=MatchValue(value=tenant_id))])`
> 
> **优点**：
> 
> - 管理简单（一个 collection）
> - 资源利用率高（共享索引）
> - 便于批量操作（清理租户数据）
> 
> **缺点**：
> 
> - 依赖过滤性能（需要索引 `tenant_id` 字段）
> - 误操作可能跨租户（需要权限控制）
> 
> **对比方案 A（独立集合）**：
> 
> - 优点：隔离最强，不会误操作
> - 缺点：集合数量多，管理成本高，资源利用率低

#### Q: 向量检索的延迟瓶颈在哪？怎么优化？

> **瓶颈分析**：
> 
> 1. **Embedding**：CLIP 推理 ~200ms（CPU），可优化
> 2. **Qdrant 检索**：~50ms（Cloud），很快
> 3. **Rerank**：DashScope API ~800ms（P95），最慢
> 
> **优化方案**（按成本递增）：
> 
> 1. **批量 Rerank**：一次传多个 query-doc 对，减少 API 调用次数
> 2. **本地 Rerank**：部署 BGE-reranker-v2-m3，避免 API 延迟和成本
> 3. **缓存**：对热门 query 的检索结果做 Redis 缓存（TTL 1 小时）
> 4. **异步化**：embedding 和 rerank 改为异步队列，避免阻塞请求
> 
> **实际效果**：
> 
> - Rerank 延迟从 3s 降到 800ms（批量调用）
> - 成本降低 70%（只 rerank top-10，不是全部候选）

#### Q: 向量相似度计算用什么距离？为什么？

> **Cosine 相似度**（归一化后的点积）
> 
> - 公式：`cos(θ) = (A · B) / (||A|| × ||B||)`
> - 范围：[-1, 1]，值越大越相似
> 
> **为什么用 Cosine**：
> 
> - 文本/图像 embedding 通常是高维稀疏向量
> - Cosine 只关心方向，不关心 magnitude（长度）
> - 适合语义相似度（方向相同 = 语义相似）
> 
> **对比其他距离**：
> 
> - Euclidean（L2）：关心绝对距离，适合坐标数据
> - Dot Product：关心 magnitude，适合推荐系统
> - Qdrant 支持三种，根据场景选择

### 7.2 Redis

#### Q: Redis 在项目中承担什么角色？

> **三个角色**：
> 
> 1. **会话持久化**（SessionManager）
>    
>    - 存储用户会话历史、偏好、摘要
>    
>    - Key 设计：`agent:tenant:{tenant_id}:session:{session_id}`
>    
>    - TTL：7 天（用户会话有效期）
> 
> 2. **LangGraph Checkpointer**（RedisSaver）
>    
>    - 存储 Agent 每步状态快照
>    
>    - Key 设计：`agent:checkpoint:{thread_id}:{checkpoint_id}`
>    
>    - 支持中断恢复、多副本共享会话
> 
> 3. **配额管理**（租户用量计数）
>    
>    - 存储每日请求数、每月 token 数
>    
>    - Key 设计：`agent:tenant:{tenant_id}:usage:req:{date}`
>    
>    - TTL：1 天/31 天（按日/月统计）

#### Q: Redis 挂了怎么办？

> **降级策略**（保证服务可用性）：
> 
> 1. **SessionManager**：回退到内存 dict（单进程可用，多进程丢会话）
> 2. **LangGraph Checkpointer**：回退到 InMemorySaver（服务重启丢状态）
> 3. **配额管理**：暂时跳过，不阻断请求（但可能超配额）
> 
> **设计原则**：
> 
> - Redis 是性能优化，不是核心依赖
> - 降级后服务可用，但功能受限
> - 监控 Redis 健康状态，及时告警

#### Q: Redis 的 Hash 和 String 的区别？为什么用 Hash 存会话？

> **String**：
> 
> - 整个值一起读写
> - 适合简单数据（计数器、缓存）
> - 示例：`SET session:123 "user_id=456,history=[...]"`
> 
> **Hash**：
> 
> - 字段级别读写
> - 适合结构化数据（对象、记录）
> - 示例：`HSET session:123 user_id 456 history [...]`
> 
> **为什么用 Hash 存会话**：
> 
> - 会话包含多个字段：history、preferences、summary、created_at
> - 可以单独更新某个字段（如只更新 history，不覆盖 preferences）
> - 可以单独读取某个字段（如只读 preferences，不加载 history）
> - 节省内存（不需要序列化整个对象）

#### Q: Redis 的 TTL 策略是什么？为什么？

> **TTL（Time To Live）策略**：
> 
> - **Session**：7 天（用户会话有效期）
>   
>   - 用户 7 天不活跃，会话自动清理
>   - 节省内存，避免无限增长
> 
> - **Quota Counter**：1 天/31 天（按日/月统计）
>   
>   - 日计数器：`usage:req:2026-07-01`，1 天后自动清理
>   - 月计数器：`usage:tok:2026-07`，31 天后自动清理
>   - 自动滚动统计，不需要手动清理
> 
> - **Checkpoint**：不设置（依赖 LangGraph 清理）
>   
>   - Checkpoint 需要长期保留（支持中断恢复）
>   - LangGraph 自带清理机制（保留最近 N 个 checkpoint）

#### Q: Redis 和 MySQL 的区别？为什么不用 MySQL 存会话？

> **Redis（内存数据库）**：
> 
> - 低延迟：读写 ~0.1ms（内存）
> - 支持 TTL：自动过期
> - 数据结构丰富：String、Hash、List、Set、Sorted Set
> - 适合：临时数据、缓存、计数器
> 
> **MySQL（磁盘数据库）**：
> 
> - 高延迟：读写 ~10-100ms（磁盘）
> - 不支持 TTL：需要手动清理
> - 支持复杂查询：JOIN、WHERE、ORDER BY
> - 适合：持久化数据、复杂查询
> 
> **为什么不用 MySQL 存会话**：
> 
> - 会话是临时数据，需要快速读写
> - 会话不需要复杂查询（只按 session_id 查询）
> - 会话需要自动过期（MySQL 需要定时任务）
> - Redis 更适合会话场景（快、简单、自动过期）

#### Q: Redis 的内存淘汰策略？

> **内存淘汰策略**（内存满了怎么办）：
> 
> - **noeviction**：不淘汰（内存满了报错）
> - **allkeys-lru**：淘汰最近最少使用的 key（不管有没有 TTL）
> - **volatile-lru**：淘汰有 TTL 的 key 中最近最少使用的
> - **allkeys-random**：随机淘汰
> - **volatile-random**：随机淘汰有 TTL 的 key
> - **volatile-ttl**：淘汰 TTL 最短的 key
> 
> **当前配置**：`volatile-lru`
> 
> - 只淘汰有 TTL 的 key
> - 优先淘汰最近最少使用的
> - 保护没有 TTL 的 key（如 checkpoint）

## 九、常见坑和注意事项

### 9.1 不要说的话

- ❌ "这是一个毕设项目" → ✅ "这是一个完整的 RAG Agent 工程实践"
- ❌ "我调用了 XXX API" → ✅ "我设计了 XXX 链路，解决了 XXX 问题"
- ❌ "指标是 XXX" → ✅ "指标是 XXX，因为 XXX 原因，我做了 XXX 优化"
- ❌ "我用了 LangChain" → ✅ "我用了 LangGraph，因为它支持状态机和 checkpointer"

### 9.2 会被追问的点

- **Qdrant 索引参数怎么调的？**
  - 答：默认参数（m=16, ef_construct=100, ef=128），根据召回率调整 ef
- **Redis 内存占用多少？**
  - 答：200 商品 ≈ 50MB（向量 1024 维 × 200 × 4 字节 ≈ 800KB，加上 payload ≈ 50MB）
- **为什么不用 Milvus？**
  - 答：Qdrant 更轻量、Python SDK 友好、命名向量原生支持

### 9.3 诚实原则

- 没做的不要说做了（如线上部署、压测）
- 指标要真实（0.82 就是 0.82，不要说 0.85）
- 优化要讲清楚 trade-off（如 rerank 延迟 vs 准确率）
- 不会的不要瞎编（可以说"这个我还没研究，但我的理解是..."）
