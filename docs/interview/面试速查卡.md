## 🎯 30 秒电梯演讲

我做了一个基于 RAG 的电商导购 Agent，用户上传图片或输入文字，系统通过 **CLIP 图像特征 + BGE-M3 文本特征**做混合检索，**千问 gte-rerank-v2** 精排后交给 **DeepSeek** 生成带引用的购物推荐。**200 个 SKU** 测试中，**同款识别 Top-1 达到 100%**，**Faithfulness 0.82**。工程上用了 **LangGraph** 做 Agent 编排、**Redis** 做会话和状态持久化、**Qdrant** 做向量存储，还实现了**多租户架构**支持多团队共用后端。

| 指标 | 数值 | 一句话解释 |

| **SKU 规模** | 200 商品 | 跨品类 PoC，验证链路正确性 |

## 🏗️ 技术架构（白板讲解）

```
用户输入（图片/文字）
    ↓
Embedding 层
├── CLIP → 512 维图像向量
└── 阿里云 text-embedding-v3 → 1024 维文本向量
    ↓
Qdrant 混合检索（payload filter: tenant_id + category）
├── 图像 top-k（score_threshold=0.6）
└── 文本 top-k（score_threshold=0.5）
    ↓
RRF 融合：score(d) = Σ 1/(60 + rank_i(d))
    ↓
千问 gte-rerank-v2 精排（top-10 → top-3）
    ↓
结构化过滤（price filter）
    ↓
LangGraph Agent 编排
├── 槽位填充（预算/品类/场景/偏好）
├── 检索节点 → 候选商品
├── 引用节点 → citation 片段
├── 澄清节点 → 多因子门控（score + gap + 槽位）
└── 生成节点 → DeepSeek 流式输出（显式引用 + 不确定表达）
    ↓
SSE 推送前端（delta_text / candidates / citations / final）
```

## 🔑 技术选型（为什么这么选）

| 组件 | 选型 | 为什么 | 替代方案 |
|------|------|--------|---------|
| 图像 Embedding | CLIP (ViT-B/32) | 图文对齐，512 维轻量 | SigLIP, BLIP |
| 文本 Embedding | 阿里云 text-embedding-v3 | 中文效果好，1024 维 | OpenAI, Cohere |
| Rerank | 千问 gte-rerank-v2 | 中文电商场景好 | BGE-reranker 本地 |
| 向量库 | Qdrant Cloud | 开源，命名向量，免运维 | Milvus, Weaviate |
| Agent 框架 | LangGraph | 状态机可控，checkpointer | LangChain Agent |
| LLM | DeepSeek | 性价比高，中文好 | 通义千问, GLM-4 |
| 会话存储 | Redis | 低延迟，TTL，checkpointer | PostgreSQL |

## 💾 数据库问题（必问）

### Qdrant

**Q: 为什么选 Qdrant？**
- 开源轻量（Rust 单二进制）
- 命名向量原生支持（text + image）
- Cloud 免运维

**Q: HNSW 索引原理？**
- 分层图结构：顶层稀疏（快速定位），底层密集（精确搜索）
- 参数：m=16, ef_construct=100, ef=128
- 调优：召回率低 → 增加 ef

**Q: 多租户隔离？**
- 方案 B：Payload Filter（共享集合 + tenant_id 过滤）
- 优点：管理简单，资源利用率高
- 缺点：依赖过滤性能，需要索引 tenant_id

**Q: 延迟瓶颈？**
- Rerank（DashScope API ~800ms P95）
- 优化：批量调用、本地模型、缓存、异步化

### Redis

**Q: 三个角色？**
1. 会话持久化（SessionManager）
2. LangGraph Checkpointer（RedisSaver）
3. 配额管理（租户用量计数）

**Q: Redis 挂了怎么办？**
- SessionManager → 内存 dict
- Checkpointer → InMemorySaver
- 配额管理 → 暂时跳过

**Q: 为什么用 Hash 存会话？**
- 字段级别读写（history、preferences、summary）
- 可以单独更新/读取某个字段
- 节省内存（不需要序列化整个对象）

**Q: TTL 策略？**
- Session: 7 天
- Quota: 1 天/31 天
- Checkpoint: 不设置（LangGraph 清理）

**Q: Redis vs MySQL？**
- Redis：内存、低延迟（0.1ms）、支持 TTL、适合临时数据
- MySQL：磁盘、高延迟（10-100ms）、支持复杂查询、适合持久化

## 🎨 技术亮点（讲述顺序）

1. **混合检索 + Rerank**（最核心）
   - CLIP + BGE-M3 双路检索
   - RRF 融合（不需要归一化）
   - Rerank 精排（延迟从 3s 降到 800ms）

2. **多租户架构**（工程化能力）
   - X-Tenant-ID 中间件
   - Redis key 隔离
   - Qdrant payload filter
   - 配额管理（请求数 + token 数）

3. **四层记忆**（系统设计）
   - Working（LangGraph state）
   - Episodic（Redis Hash - 会话历史）
   - Semantic（Redis Hash - 用户偏好）
   - Perceptual（Qdrant - 图像/文本向量）

4. **LLM 治理**（生产环境意识）
   - Fallback（DeepSeek → 千问）
   - 熔断（失败率过高快速失败）
   - 重试（指数退避 3 次）
   - 限流（令牌桶 10 req/s）

## ⚠️ 面试坑点

### 不要说
- ❌ "毕设项目" → ✅ "完整的 RAG Agent 工程实践"
- ❌ "调用了 API" → ✅ "设计了链路，解决了问题"
- ❌ "指标是 XXX" → ✅ "指标是 XXX，因为 XXX，我做了 XXX 优化"

### 会被追问
- Qdrant 索引参数怎么调？→ 默认参数，根据召回率调整 ef
- Redis 内存占用？→ 200 商品 ≈ 50MB
- 为什么不用 Milvus？→ Qdrant 更轻量、Python SDK 友好

### 诚实原则
- 没做的不要说做了（线上部署、压测）
- 指标要真实（0.82 就是 0.82）
- 不会的不要瞎编（"这个我还没研究，但我的理解是..."）
